# 语音仿读质量评估系统 — UI 需求书

> 版本：v1.0 | 日期：2026-07-09 | 状态：需求评审

---

## 1. 项目背景与目标

### 1.1 现有系统概述

语音仿读质量评估系统是一个面向英语教学场景的自动化评估工具。系统从**语音相似度**（OpenSMILE 8 个声学维度）和**文字准确度**（Whisper 转写 + LLM 逐句比对）两个维度，对学生的英语仿读录音进行量化评分，并生成可视化报告。

当前系统通过命令行运行 8 阶段流水线，所有配置通过 `config.yaml` 和 `.env` 文件手动编辑，输出为静态文件（PNG / CSV / Excel / Markdown / JSON）。系统没有图形界面，使用者需要直接操作文件系统和配置文件。

### 1.2 UI 项目目标

构建一个**包装现有命令行程序**的图形用户界面，实现以下核心目标：

| 目标 | 说明 |
|------|------|
| **降低使用门槛** | 教师无需接触命令行和配置文件即可完成评估全流程 |
| **提升操作效率** | 可视化文件管理、一键启动、实时进度反馈，减少重复操作 |
| **增强结果洞察** | 交互式图表替代静态 PNG，支持筛选、排序、下钻分析 |
| **支撑未来扩展** | 架构预留多用户、多角色扩展能力，与 TODO 路线图对齐 |

### 1.3 设计原则

1. **零侵入集成**：不修改现有 `src/` 业务逻辑代码，通过薄封装层调用现有模块 API
2. **美观优先**：采用现代化设计语言，重视视觉一致性和交互细节
3. **渐进演进**：Phase 1 单用户本地应用 → Phase 2 师生双端 → Phase 3 校师生三方平台，每阶段都是前一阶段的扩展而非重写
4. **命令行兼容**：`python run.py` 控制台模式持续可用，UI 作为可选启动方式（`python run_ui.py`）

## 2. 用户画像与使用场景

### 2.1 目标用户

| 角色 | 描述 | 技术能力 | 核心需求 |
|------|------|---------|---------|
| **教师 / 研究者** | 英语教师或语音研究者，负责评估学生仿读质量 | 熟练使用办公软件，不熟悉命令行 | 批量评估、查看结果、追踪进步 |
| **学生**（Phase 2） | 提交仿读录音的学习者 | 仅需上传录音和查看个人反馈 | 提交录音、查看个人成绩和错误标注 |

### 2.2 核心工作流

```
准备素材 → 配置参数 → 启动评估 → 监控进度 → 查看结果 → 分析错题 → 追踪历史 → 导出报告
```

| 阶段 | 用户动作 | 频率 |
|------|---------|------|
| **准备** | 上传标准音频/文本，批量导入学生录音 | 每次新课 / 每周 |
| **配置** | 调整 LLM 模型、模块开关、评分权重 | 偶尔 |
| **评估** | 一键启动，实时查看各阶段进度 | 每次上课后 |
| **查阅** | 浏览总览表，点击学生查看详情 | 每次评估后 |
| **分析** | 查看词云和错误分布，识别共性薄弱点 | 阶段性 |
| **追踪** | 对比历史批次，查看个人进步曲线 | 期中/期末 |
| **导出** | 下载 Excel 汇总、打印个人报告 | 按需 |

## 4. 非功能性需求

### 4.1 性能

| 指标 | 要求 |
|------|------|
| 页面加载 | 首屏 < 2 秒（本地网络） |
| 大文件上传 | 支持 > 100MB 音频，显示上传进度条 |
| 长任务响应 | 评估运行期间 UI 不阻塞，可切换页面 |
| 表格渲染 | 支持 500+ 学生数据流畅滚动（虚拟滚动） |
| 图表渲染 | 单页多图表懒加载，首屏 3 秒内可交互 |

### 4.2 美观

- **设计语言**：采用现代简约风格，参考 Notion / Linear 的克制配色和间距体系
- **色彩系统**：
  - 主色：学术蓝（#2563EB），传达专业可信赖
  - 语义色：成功绿 / 警告黄 / 错误红，用于评分等级和错误标注
  - 中性色：冷灰阶（#F8FAFC → #0F172A），构建清晰的视觉层级
- **字体**：中文界面使用系统默认无衬线字体（Windows: Microsoft YaHei, macOS: PingFang SC），英文数据使用等宽字体（JetBrains Mono 或 Consolas）
- **响应式**：适配 1280px–2560px 宽度的桌面屏幕
- **暗色模式**：支持浅色/深色主题切换，自动跟随系统设置
- **动画**：页面过渡、进度条、图表加载使用微动画（duration ≤ 300ms），不干扰操作

### 4.3 可用性

- **界面语言**：全中文（功能标签、提示文本、错误消息），代码/数据字段保留英文
- **操作引导**：首次使用展示 3-5 步引导流程（上传标准素材 → 导入学生音频 → 配置参数 → 启动评估）
- **错误提示**：表单校验即时反馈、文件格式不合规具体说明、API 异常提供排查建议
- **确认机制**：删除数据、覆盖文件、取消任务等不可逆操作需二次确认
- **键盘快捷键**：常用操作支持快捷键（如 Ctrl+Enter 启动评估、Ctrl+S 保存配置）

### 4.4 可靠性

- **断点续传**：崩溃或主动停止后，下次启动可识别未完成任务并从断点继续
- **错误隔离**：单个学生处理失败不影响其他学生的评估和 UI 整体稳定性
- **数据持久化**：所有评估结果在本地文件系统完整保存，不因 UI 关闭而丢失
- **日志记录**：所有操作和异常写入日志文件，支持问题回溯

### 4.5 扩展性

- **前后端分离**：通过 RESTful API 通信，前端不依赖后端实现细节
- **API 版本化**：`/api/v1/...`，后续版本向后兼容
- **模块化前端**：UI 组件独立封装，新增分析模块只需增加路由和组件
- **插件化配置**：分析维度的增删通过配置文件驱动，无需改动前端代码

## 5. 技术架构建议

### 5.1 推荐方案：FastAPI + Vue 3（本地 Web 应用）

采用 **Python FastAPI 后端 + Vue 3 前端** 的 Web 架构，浏览器作为 UI 容器，`python run_ui.py` 一键启动本地服务器并自动打开浏览器。

| 层 | 技术选型 | 职责 |
|----|---------|------|
| **后端框架** | FastAPI（Python） | REST API + SSE 实时推送，异步 I/O |
| **前端框架** | Vue 3 + TypeScript | 组件化 UI，响应式数据绑定 |
| **UI 组件库** | Element Plus 或 Naive UI | 表格/表单/对话框/通知等标准组件 |
| **图表库** | ECharts 5 | 雷达图、折线图、直方图、词云 |
| **构建工具** | Vite | 前端开发和打包 |
| **打包分发** | PyInstaller（内嵌 Python + 静态前端） | 一键式桌面应用 |

### 5.2 方案对比

| 维度 | **FastAPI + Vue 3** ★推荐 | PyQt6 | Electron | Gradio |
|------|---------------------------|-------|----------|--------|
| UI 美观度 | ★★★★★ 现代 Web 设计 | ★★★☆☆ 受限于 QSS | ★★★★★ 现代 Web 设计 | ★★☆☆☆ 模板化 |
| 开发效率 | ★★★★☆ 前后端分离 | ★★★★☆ Python 全栈 | ★★★☆☆ 需 JS 全栈 | ★★★★★ 极快 |
| 扩展性（多用户） | ★★★★★ 天然支持 | ★☆☆☆☆ 需完全重写 | ★★★★☆ 前端可复用 | ★★☆☆☆ 受限 |
| 打包体积 | ★★★★★ 零（浏览器） | ★★★☆☆ ~200MB | ★★☆☆☆ ~300MB | ★★★☆☆ ~500MB |
| 文件系统访问 | ★★★★★ Python 直连 | ★★★★★ 原生直连 | ★★★☆☆ Node IPC | ★★★★★ Python 直连 |
| 实时更新 | ★★★★★ SSE 原生异步 | ★★★★☆ QThread 信号 | ★★★★☆ IPC | ★★★☆☆ 轮询为主 |
| 学习成本（Python 团队）| ★★★★☆ 低（两边都是 Python 生态）| ★★★★★ 最低 | ★★☆☆☆ 需学 JS/TS | ★★★★★ 最低 |

### 5.3 集成策略

核心原则：**不修改现有 `src/` 模块，通过薄封装层衔接 UI 与业务逻辑。**

```
src/                    ← 现有业务逻辑（不动）
  launcher.py
  voice_compare.py
  text_llm.py
  ...

services/              ← 新增 UI 服务层（薄封装）
  pipeline_service.py   — 封装 launcher 调用，注入进度回调
  file_service.py       — 封装文件上传、校验、管理
  config_service.py     — 封装配置读写
  result_service.py     — 封装结果查询和聚合
  export_service.py     — 封装导出逻辑

server/                 ← 新增 Web 层
  main.py               — FastAPI 应用入口
  api/v1/               — REST 路由
  sse/                  — 实时事件推送

ui/                     ← 新增前端（独立目录）
  src/
    views/              — 页面视图
    components/         — 可复用组件
    stores/             — 状态管理
    api/                — API 调用封装
```

### 5.4 关键集成点

- **进度事件流**：在 `launcher.py` 各阶段插入可选回调 `progress_cb(stage, current, total, status)`，UI 通过 SSE 订阅
- **配置双向同步**：扩展 `config.py` 增加 `save()` 方法，UI 修改后写回 `config.yaml`
- **文件上传**：`file_service.py` 负责接收前端上传的文件并写入 `resource/` 对应目录，含命名校验
- **Whisper 模型预热**：服务启动时加载 Whisper 模型到内存，避免首次任务等待过久

## 6. 分阶段实施路线图

### Phase 1：单用户本地桌面应用（当前阶段）

**目标**：完整包装现有功能，提供美观高效的本地 UI。

- 覆盖 3.1–3.9 全部功能模块
- 本地启动，单用户，无需登录
- 文件系统直接读写（不引入数据库）
- 浏览器访问 `http://localhost:8000`
- 打包为 PyInstaller 桌面应用（可选）

### Phase 2：师生双端

**目标**：引入用户系统，实现教师管理、学生提交的协作模式。

- 新增用户系统（注册/登录/JWT 认证）
- 教师端：创建班级、发布任务、查看全班结果
- 学生端：查看任务、上传录音、查看个人报告
- 数据持久化：SQLite（单机部署）或 PostgreSQL（服务器部署）
- 前端增量开发（复用 Phase 1 80% 组件）

### Phase 3：校师生三方平台

**目标**：面向全校的多角色平台，支持大规模并发评估。

- 管理员后台：学校管理、教师管理、系统配置
- RBAC 多角色权限体系
- 消息通知（任务发布、评估完成）
- Docker 容器化部署
- PostgreSQL + Redis 缓存层
- API 限流、审计日志

## 7. 风险与约束

### 7.1 技术约束

| 约束 | 来源 | 影响 | 应对策略 |
|------|------|------|---------|
| OpenSMILE C++ 库非线程安全 | voice_compare.py | 无法并行分析多个学生的语音 | 后端以单 worker 队列序列化语音任务 |
| Whisper 模型加载慢（10-30 秒）| text_llm.py | 首次使用等待久 | 服务启动时预热加载，内存常驻 |
| matplotlib 必须用 Agg 后端 | voice_compare.py | 多线程下 Tkinter 崩溃 | FastAPI 进程级别天然隔离，无冲突 |
| LLM API 超时/限流 | DeepSeek API | 批量评估时部分学生失败 | 重试机制 + 失败不阻塞 + 单学生重试入口 |
| Windows 路径与编码 | 操作系统 | 中文文件名和路径可能异常 | 统一 UTF-8，路径处理用 pathlib |

### 7.2 项目风险

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 前后端开发技能要求 | 中 | 前端选用 Vue 3 + 成熟组件库降低学习曲线；初始版本可由一人完成 |
| OpenSMILE 版本兼容性 | 低 | 固定 OpenSMILE 版本号，虚拟环境管理依赖 |
| 功能范围蔓延 | 中 | 严格按 Phase 优先级开发，Phase 1 聚焦核心 P0 功能 |
| 打包分发体积过大 | 低 | Web 架构本身零打包；PyInstaller 可选，通过 CDN 加载前端资源 |

### 7.3 设计约束

- **当前仅支持 Windows**：OpenSMILE 在 Windows 上安装配置最为成熟，跨平台适配留待 Phase 2 评估
- **OpenAI 兼容 API 依赖**：LLM 比对功能依赖 OpenAI 格式的 Chat Completions API
- **浏览器依赖**：Web 架构要求系统有现代浏览器（Edge/Chrome/Firefox），Windows 11 自带 Edge，不构成额外要求
